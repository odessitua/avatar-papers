# Техническое задание: скрипт сбора статей и публикации в Confluence

## 1. Анализ источника и ключевые слова

### 1.1 Референсный репозиторий

- **URL:** https://github.com/liutaocode/talking-face-arxiv-daily  
- **Назначение:** автоматический сбор статей с arXiv по заданным ключевым словам (Talking Face, Lip Sync и т.д.), формирование README и веб-страницы.

### 1.2 Как устроен сбор у них

- Источник данных — **официальный API arXiv** (поиск по запросу, сортировка по дате публикации/обновления).
- Ключевые слова задаются в `config.yaml` в виде списков по темам. Из конфига формируется поисковый query (слова объединяются через OR).
- Результаты сохраняются в JSON, из JSON генерируется Markdown (README, docs для GitHub Pages).
- Ссылка на код: в их пайплайне предусмотрен поиск репозитория на GitHub (по title/arxiv id), но в текущей версии этот шаг отключён (SSL/внешние запросы). Идею можно повторить у себя.

### 1.3 Ключевые слова для парсинга (взять из референса)

Использовать те же группы ключевых слов, что в [config.yaml](https://raw.githubusercontent.com/liutaocode/talking-face-arxiv-daily/main/config.yaml):

**Тема «Talking Face»** (объединять через OR в одном запросе к arXiv):

- Talking Face, Talking Head, Talking Avatar, Animate Portrait, Visual Dubbing  
- Face Generation, Lip Sync, Talker, Portrait, Talking Video  
- Head Synthesis, Face Reenactment, Wav2lip, Lip Generation, Lip Synchronization  
- Portrait Animation, Facial Animation, Lip Expert  

**Тема «Image Animation»:**

- Image Animator, Image Animation  

В конфиге скрипта должна быть возможность задать свои ключевые слова и при необходимости добавить новые темы (аналогично структуре `keywords` в референсе).

### 1.4 Независимость от репозитория

- Не использовать их JSON/README как источник списка статей.
- Реализовать свой запрос к **arXiv API** по указанным ключевым словам (те же или свои), свою логику сортировки и лимитов (например, по дате обновления, max_results на тему).
- При желании можно добавить опцию «подтянуть список из их JSON» как дополнительный режим, но основной сценарий — только arXiv API.

## 2. Таблица статей

### 2.1 Назначение

Единая таблица для учёта статей: что найдено, что скачано, что обработано, куда опубликовано. По ней скрипт решает, что качать, а пользователь/Cursor — что отдавать на анализ и публикацию.

### 2.2 Формат и расположение

- **Формат:** CSV.
- **Имя файла:** например `papers.csv` (путь задаётся в конфиге, например корень проекта или папка `data/`).

### 2.3 Колонки таблицы

| Колонка       | Тип/формат        | Описание |
|---------------|-------------------|----------|
| arxiv_id      | строка            | Идентификатор arXiv (например 2603.01418). |
| url           | строка            | Ссылка на статью (https://arxiv.org/abs/{arxiv_id}). |
| title         | строка            | Название статьи. |
| date          | строка/дата       | Дата публикации или последнего обновления (единый формат, например YYYY-MM-DD). |
| authors       | строка            | Авторы (кратко, например «First Author et al.»). |
| code_url      | строка            | Ссылка на код (репозиторий), если найдена; иначе пусто. |
| downloaded    | булево/да-нет     | PDF скачан в папку оригиналов (да/нет или 1/0). |
| processed     | булево/да-нет     | Статья обработана: есть файл анализа в папке вывода LLM (да/нет или 1/0). |

При необходимости можно добавить колонку `topic` (Talking Face / Image Animation и т.д.) для фильтрации и отчётов.

---

## 3. Файловая структура

### 3.1 Папки

- **Оригиналы статей (PDF)**  
  - Одна папка, например `papers/originals/` или `pdf/`.  
  - Имя файла: `{arxiv_id}.pdf` (например `2603.01418.pdf`).  
  - Скачивание только через скрипт; при успешной загрузке в таблице выставляется `downloaded = да`.

- **Выход анализа LLM**  
  - Одна папка, например `papers/analysis/` или `analysis/`.  
  - Один файл на статью, например `{arxiv_id}.md` или `{arxiv_id}.txt`.  
  - Содержимое — по шаблону, который задаёт пользователь при обращении к Cursor (см. п. 5).  
  - Скрипт не генерирует эти файлы; при появлении файла (или при явном указании в логике) можно обновлять в таблице `processed = да`.

### 3.2 Конфиг

- Конфигурация скрипта (ключевые слова, пути к папкам, параметры arXiv, настройки Confluence) хранится в конфиг-файле (например YAML или .env).  
- Пути к таблице, папке оригиналов и папке анализа задаются в конфиге.

## 4. Функции скрипта (поиск и скачивание)

### 4.1 Поиск статей

- Читать ключевые слова из конфига (по темам, как в п. 1.3).
- Для каждой темы (или для объединённого запроса) выполнять запрос к **arXiv API** (библиотека `arxiv` или прямой HTTP к api.arxiv.org).
- Параметры: query из ключевых слов, сортировка по дате (published или updated), ограничение числа результатов (max_results на тему или общий лимит).
- Для каждой найденной статьи: arxiv_id, title, date, authors, url. По желанию — попытка определить ссылку на код (см. п. 4.3).

### 4.2 Обновление таблицы

- Добавлять в таблицу только **новые** записи (по arxiv_id), не дублировать.
- Обновлять у существующих записей при необходимости: title, date, authors, code_url (если появилась ссылка на код).
- Колонки `downloaded` и `processed` обновляются отдельно: `downloaded` — после успешного скачивания PDF, `processed` — по факту наличия файла в папке анализа или по явной команде.

### 4.3 Ссылка на код (опционально)

- Для каждой статьи пытаться получить ссылку на код:  
  - поиск на GitHub по arxiv_id или по title (как в референсном репо — `get_code_link`);  
  - или запрос к Papers With Code API по arxiv_id (если доступен и стабилен).  
- Результат записывать в колонку `code_url`; если не найдено — оставлять пустым.

### 4.4 Скачивание PDF

- Для записей с `downloaded = нет` (или без скачанного файла) скачивать PDF с arXiv  
  - URL: `https://arxiv.org/pdf/{arxiv_id}.pdf`.  
- Сохранять в папку оригиналов с именем `{arxiv_id}.pdf`.  
- При успешном сохранении выставлять в таблице `downloaded = да`.  
- Обработка ошибок: логировать, не падать на одной статье; при повторном запуске докачивать только недостающее.

## 5. Обработка статей (анализ LLM)

- **Скрипт не выполняет** вызов LLM и не создаёт файлы в папке анализа.
- Пользователь вручную обращается к **Cursor** (или другому инструменту) с инструкцией вида:  
  «Вот таблица статей; возьми записи с такой-то даты / необработанные (processed = нет); для каждой по такому-то шаблону проанализируй статью (по PDF или по абстракту) и создай файл в папке `papers/analysis/` с именем `{arxiv_id}.md` и твоими ответами».
- Шаблон анализа (вопросы, структура ответа) задаётся пользователем в промпте; при желании его можно вынести в отдельный файл-шаблон в репозитории.
- После появления файлов анализа скрипт (или отдельная команда/шаг) может обновлять в таблице `processed = да` для соответствующих arxiv_id.

## 6. Публикация в Confluence

### 6.1 Назначение

- Публиковать обработанные за период (например за неделю) статьи в Confluence: одна страница на период, с форматированием (заголовки, таблицы, ссылки на статьи и код).

### 6.2 Режим: одна страница на неделю (создать или обновить)

- Логика: **одна страница Confluence соответствует одной календарной неделе** (например «Статьи за 2026-W10» или «2026-03-04 — 2026-03-10»).
- При каждом запуске публикации скрипт определяет **текущую неделю** (или неделю, заданную аргументом/конфигом) и:
  - если страницы для этой недели **нет** — **создаёт** новую;
  - если страница **уже есть** — **обновляет** её контентом по данным из таблицы и файлов анализа за эту неделю.
- То есть страница недели не дублируется: первый запуск за неделю — создание, последующие — обновление той же страницы.
- В конфиге задаётся шаблон названия страницы (например по номеру недели и году или по диапазону дат), Space Key и родительская страница (id или key) для создания дочерних.

### 6.3 Содержимое страницы

- Заголовок страницы (по шаблону или фиксированный).
- Таблица или список статей за выбранный период с колонками:  
  **дата | название | авторы | ссылка на статью (PDF/abs) | ссылка на код (если есть) | краткий вывод/тезисы** (из файла анализа, например первое предложение или отдельное поле).
- При необходимости — краткое введение (например «Обработано статей: N за период …»).

Формат контента для Confluence — **Confluence Storage Format** (XHTML-подобный) или Markdown, в зависимости от возможностей API и версии Confluence; конвертация из локального Markdown/HTML — в скрипте.

### 6.4 Доступ к Confluence

- Использовать **REST API Confluence** (создание/обновление страницы).
- Учётные данные (логин + API-токен или пароль) пользователь задаёт позже (конфиг или переменные окружения, без хардкода секретов в коде).
- В конфиге указываются: базовый URL Confluence, Space Key, родительская страница (id или key) для создания дочерних страниц по неделям.

### 6.5 Когда запускать публикацию и что попадает на страницу

- Отдельная команда или шаг скрипта: «опубликовать в Confluence обработанные статьи за [неделю]».  
- Период по умолчанию — текущая календарная неделя; при необходимости — аргумент или конфиг (неделя X, диапазон дат).  
- **Что попадает на страницу:** только записи из таблицы, у которых `processed = да` и дата статьи попадает в выбранную неделю. Контент страницы формируется из полей таблицы (url, title, authors, code_url, date) и тезисов из файлов анализа в папке вывода.

## 7. Итоговый сценарий использования

1. **Настройка:** в конфиге заданы ключевые слова (из референса или свои), пути к таблице, папкам и Confluence (URL, Space, родитель/страница для обновления); учётные данные Confluence — отдельно (env/секреты).
2. **Запуск сбора:** скрипт ищет статьи в arXiv по ключевым словам, обновляет таблицу (новые строки, при необходимости code_url), скачивает недостающие PDF в папку оригиналов, выставляет `downloaded`.
3. **Обработка:** пользователь открывает таблицу и в Cursor даёт задание: выбрать статьи (по дате / необработанные), проанализировать по шаблону, записать результат в папку анализа; при необходимости скрипт или пользователь выставляет `processed`.
4. **Публикация:** пользователь запускает шаг «опубликовать в Confluence за период …»; скрипт собирает данные из таблицы и файлов анализа за период, формирует контент страницы и создаёт/обновляет страницу в Confluence через API.

---

## 8. Запуск по расписанию (cron)

- **Сбор (поиск + скачивание):** рассчитан на запуск по cron (например раз в день или раз в 12 часов). Скрипт добавляет в таблицу новые статьи по ключевым словам и докачивает PDF для записей с `downloaded = нет`. Идемпотентность: повторный запуск не дублирует строки и не перезаписывает уже скачанные файлы.
- **Публикация в Confluence:** тоже рассчитана на запуск по cron (например раз в неделю в конце недели или раз в день). Скрипт по полям таблицы определяет, какие статьи за текущую (или заданную) неделю имеют `processed = да`, формирует контент и создаёт или обновляет страницу этой недели в Confluence.
- Оба шага можно вешать на разные расписания в cron (или в GitHub Actions / другом планировщике); что публиковать — однозначно определяется полями таблицы (дата, `processed`) и выбранной неделей.
